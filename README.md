# Semantic Search Migration Plan
### Backend Embedding (fastembed) + Local Cosine Search

---

## Summary

We are migrating semantic search from a system that **runs a 30MB AI model on the popup's main thread (causes freeze)** to one that **calls a backend `/api/embed` endpoint on every memory save, stores the embedding locally, and searches with a cosine for-loop**. The popup does zero AI computation. No UI freeze.

---

## The Problem

The `VectorMemoryClassifier` class in `popup.js` (lines 181–930) downloads and runs `all-MiniLM-L6-v2` on the **popup's main UI thread**. Chrome's CSP blocks Web Workers in extension popups, so all AI computation blocks the UI.

**Result:** Complete UI freeze while model downloads on frontend.

---

## Solution

> **Embeddings are generated by the backend. Search happens locally.**

The popup becomes a thin UI layer. It never touches any AI model or complex vector math.

### Flow: Memory Saved
```
Memory is saved
  ↓
MemoryDB.addMemory(memory) called  
  ↓
addMemory calls POST /api/embed with memory text
  ↓
Backend (fastembed) returns 384-dim vector
  ↓
memory.embedding = [...] attached silently
  ↓
{ text, embedding, timestamp, platform, ... } stored in IndexedDB
  ↓
User only sees the text — embedding is invisible
```

### Flow: User Searches
```
User types query (300ms debounce)
  ↓
EmbeddingClient.search(query, allMemories) called
  ↓
Calls POST /api/embed with query text
  ↓
Backend returns query embedding
  ↓
JS for-loop: cosineSimilarity(queryEmbedding, memory.embedding) for each memory
  ↓
Sort by score → return top K → render results
  ↓
Zero freeze — all heavy math is one API call + 1ms of local arithmetic
```

### For Loop is fast enough for Now:
```
900 memories × 384 dimensions = 345,600 operations
V8 engine: ~500M ops/sec → < 1ms
```
No vector database (FAISS, pgvector) needed at this scale. Maybe if we expand to 100k+ memories, then we definetly need a vectorDB like FAISS or pgvector, for now its overengineering and unnecessary.

---

No data is sent to Google, OpenAI, or any third party.

```javascript
// Before:
async addMemory(memory) {
  await this.init();
  return new Promise((resolve, reject) => {
    const request = store.put(memory); // ← no embedding
    ...
  });
}

// After:
async addMemory(memory) {
  await this.init();

  // Auto-embed before saving
  if (!memory.embedding && (memory.content || memory.text)) {
    try {
      const text = memory.content || memory.text;
      const res = await fetch('https://memory-box-website.onrender.com/api/embed', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        credentials: 'include',
        body: JSON.stringify({ text })
      });
      if (res.ok) {
        const data = await res.json();
        memory.embedding = data.embedding; // silent, invisible to user
      }
    } catch (e) {
      // Non-fatal: saved without embedding, falls back to keyword search
      console.warn('⚠️ Embed failed, saving without:', e.message);
    }
  }

  return new Promise((resolve, reject) => {
    const request = store.put(memory); // ← now includes embedding
    ...
  });
}
```

Everything else in `memoryDB.js` is unchanged. No schema migration needed — `embedding` field already exists.

## API Contract

### `POST /api/embed`

| | |
|--|--|
| **URL** | `https://memory-box-website.onrender.com/api/embed` |
| **Auth** | Session cookie (`credentials: 'include'`) |

**Request:**
```json
{ "text": "I prefer morning meetings over afternoon ones" }
```

**Response (200):**
```json
{
  "embedding": [0.013, -0.82, 0.441, ...],
  "dimensions": 384,
  "latency_ms": 45.2
}
```

**Error responses:**

| Code | Reason |
|------|--------|
| 400 | Missing or empty text |
| 400 | Text > 10,000 characters |
| 401 | Not authenticated |
| 503 | Model not loaded |
| 500 | Unexpected error |

## Data Migration Strategy 

Existing memories in IndexedDB have either:
- `embedding: null` — needs embedding
- `embedding: [...]` (384-dim) — already compatible (same model: `all-MiniLM-L6-v2`)

**On first launch after update:**
1. Detect memories with `embedding === null`
2. Call `embeddingClient.batchEmbed()` silently in the background
3. Show subtle progress: *"Upgrading search... (47/200)"*
4. Memories without embeddings fall back to keyword search during migration
5. No user action required

## Risk Register

| Risk | Mitigation |
|------|-----------|
| `fastembed` OOM on Render | fastembed is ~150MB (vs PyTorch ~900MB) — should fit comfortably |
| `/api/embed` call adds latency to memory save | Save is async; user already saves memory while embed runs in background |
| Network failure during embed | Catch block saves memory without embedding; falls back to keyword search |
| Rate limit exceeded by active user | 300ms debounce on search + 60 req/min server-side limit |

---

## Not needed for now - dont use these until we have crazy scale:

- ❌ FAISS, pgvector, Pinecone: unnecessary at current memory scale
- ❌ Streaming embeddings or batch server-side indexing
- ❌ GPU acceleration — CPU is sufficient for single-request inference with fastembed
